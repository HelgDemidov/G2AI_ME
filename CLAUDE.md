# G2AI_ME — Claude Code memory

## О проекте
Независимый исследовательский трек. Две параллельные, но связанные линии работы:

1. **G2AI-корпус** (`sources/`, `pipeline/scripts/`) — сбор и структурирование международного опыта государств по построению политики/инфраструктуры/регулирования под GenAI и агентные сервисы («государство как MCP-сервер»), в фокусе малые страны, но не исключительно. Конечная цель — пакет предложений от независимого эксперта/отраслевой ассоциации, включая черновик Стратегии ИИ Черногории до 2030, пока правительство её разрабатывает.
2. **Более широкое видение цифрового суверенитета Черногории** (`docs/project_core/vision_and_plan.md`, v03) — параллельный (не строго вложенный в G2AI) трек: ядро локали `cnr`, мобильные клавиатуры, Wikipedia/Wikidata grounding, B2B2G с телеком-группами, дальняя цель — партнёрство государства с фронтир-лабой по «исландскому пути». G2AI-корпус — один из аналитических входов для этого видения, но не единственный и не главный по объёму документа.

Обе линии объединяет забота о точной и полной представленности Черногории в цифровом/ИИ-пространстве.

**`.gitignore` под `/sources/` — deny-by-default:** игнорируется всё (документы `raw.*`/`doc.md`/`.state.yaml`, `candidates.yaml`), КРОМЕ `meta.yaml` — курируемая мета (библиография/аналитика, не тела документов) версионируется в git, а сами документы (потенциально авторское право, см. «Статус» ниже) остаются локально. `docs/` (вся документация: видение, чартеры, спеки, бэклог) — тоже gitignored. `.claude/` — тот же паттерн deny-by-default с негейтом (PR #27): `/.claude/*` игнорируется, КРОМЕ `!/.claude/skills/` — канон Agent Skills трекается, остальное (`commands/` шимы, `settings.local.json`) остаётся локальным.

`.venv/` в КОРНЕ репо — локальный venv через `uv` (в `.gitignore`, не коммитится; venv непереносим — не копировать, а пересоздавать: `uv venv .venv && uv pip install -r requirements-dev.txt`; классический `source .venv/bin/activate` у uv-venv может не работать — вызывать бинарники как `.venv/bin/…`). Конфиг инструментов — `pyproject.toml` (mypy/ruff/pytest; `pythonpath`/`mypy_path` указывают на `pipeline/scripts`, поэтому проверки запускаются ИЗ КОРНЯ, кеши тоже оседают в корне). Проверки перед коммитом изменений в коде пайплайна (из корня, те же, что в CI): `.venv/bin/ruff check pipeline/scripts && .venv/bin/python -m mypy pipeline/scripts && .venv/bin/python -m pytest -m "not model and not corpus" && .venv/bin/python pipeline/scripts/validate_sources.py` (именно так фильтрует CI; `validate_sources.py` — 4-й шаг CI с PR #5). ⚠ Бездумный bare `pytest` без фильтра — распространённая локальная ошибка (найдена живьём 2026-07-21): тянет `@pytest.mark.corpus`-голден (`test_convert_golden.py`, реконверсия каждого документа `sources/` байт-в-байт) — ~90-100 c на 6 документах и растёт с корпусом, при этом CI её не видит вовсе. Не баг/паразитный процесс — тест легитимен, просто не «полный гейт по умолчанию»; гонять `-m corpus` осознанно при правках convert-слоя. mypy — РЕКУРСИВНО по каталогу (с repo-layout 2026-07-16 проверяются и подпакеты, и тесты; прежний file-glob `pipeline/scripts/*.py` тесты не видел вовсе). **Property-based тестирование (`hypothesis`) — общепроектный стандарт** (не разово, спек `test-coverage-hardening`, 2026-07-21) для геометрического/интервального кода — уже применён к `detect_columns`/`group_into_lines` (pdf) и `_ownership_ranges`/`_blank_foreign_cells` (xlsx); `reportlab` — dev-зависимость для синтетических PDF-фикстур в тестах (реальные объекты `pdfplumber`, генерируются на лету, не коммитятся). **Unit-тесты герметизированы от боевых артефактов** (PR #25): autouse-guard в `tests/unit/conftest.py` снимает stat до/после каждого теста с corpus.db/candidates.yaml/курсоров — тест, зовущий `main()`/CLI без явных tmp-путей (`--db`/`--root`), падает громко (живой инцидент 2026-07-21: два main()-теста без `--db` тихо вычищали боевой индекс при каждом прогоне pytest; CI не ловил — без bge-токенизатора стадия индекса пропущена).

## Пайплайн сбора документов
1. **Discovery + триаж релевантности (ДО добычи)** — WebSearch: найти реальный документ/издателя/дату (seed-список `small_states_v01.md` НЕнадёжен — реальные кейсы: фиктивные заголовки ОАЭ и Эстонии) и оценить релевантность. Кандидат заводится в `sources/candidates.yaml` (`CandidateRecord`); допущенный триажем промоутится в курируемый `meta.yaml` папки-документа через `schema.promote_candidate()`. НЕ WebFetch для тела документа — прогоняет через маленькую модель, не verbatim; годится только для метаданных. Ядро генератора кандидатов (`discovery-core`) и ручной/directed-каналы (`discovery-manual`) реализованы — см. «Слой знаний» ниже; кандидат заводится ТОЛЬКО через `discover.py inject` (единственная дверь: dedup+провенанс by construction), триаж — батч-сессией `discover.py worksheet` → decisions.yaml → `discover.py apply`; протокол поисковых кампаний — скилл `.claude/skills/directed-search/{SKILL.md,README.md}` (Agent Skill, трекается — PR #27).
2. **Скачивание оригинала** — через `run_pipeline.py`: acquisition-лестница `direct → official_alt → manual → archive` при WAF-блоке/мёртвом URL (см. «Слой знаний»), в `sources/{track}/{entity}/{doc-id}/raw.*` (расширение — `SourceFormat` записи: pdf/html/docx/xlsx); sha256/канал добычи → `.state.yaml`.
3. **Конвертация** — `converters.resolve_converter(raw)` диспетчеризует по расширению (`SourceFormat`: pdf/html/docx/xlsx) → `doc.md`; frontmatter порождается из `meta.yaml` (`schema.render_frontmatter`), не пишется вручную. Подробности — раздел «Слой конвертации» ниже.
4. **Подтверждение** — Стадия 2: по тексту `doc.md` подтвердить/понизить `relevance.target_fit`; `validate_sources.py` требует `relevance` у каждой записи корпуса (гейт в CI и в `run_pipeline`).

Подробности слоя метаданных/графа/поиска — раздел «Слой знаний» ниже; слоя конвертации (PDF/графика/HTML/OCR) — раздел «Слой конвертации» ниже.

## Слой знаний: метаданные, граф, семантический поиск
Поверх собранных `.md` — слой для кросс-документного анализа (Фазы 1-4 реализованы, влиты в `main` через PR #1; проектный план и статус — `docs/pipeline/knowledge/tech_specs/knowledge-graph-metadata/spec.md`, gitignored). Всё CPU-only, встроенное, без серверов/JVM/CUDA. Скрипты запускаются ИЗ КОРНЯ: `.venv/bin/python pipeline/scripts/<script>.py`.

- **Метаданные** — `schema.py` (слим по corpus-layout-v2, PR #4): курируемое ядро `SourceRecord` (~19 полей) в `meta.yaml` — id, `entity_id`/`track`, issuer_type, geo_scope, language (ISO 639-1, либо 639-3 там, где 639-1 нет — черногорский `cnr`; PR #5), dates, doc_type, authority, source_url, sensitivity, rights, topics, g2ai_pattern, summary, relations, relevance (Dublin Core + минимум аналитики). Операционное (sha256/acquisition/…) — отдельная модель `OperationalState` в `.state.yaml`-sidecar (машина пишет `save_state` через `fsio.atomic_write_text`, НЕ в `meta.yaml`; comment-safe); `raw_size`/`raw_mtime_ns` — stat-guard, чтобы не перечитывать raw целиком на каждом прогоне (PR #7). Инварианты раскладки (папка==id, entity_id, track; глоб. уникальность id) — единая `schema.check_layout()` (PR #8), используется и `load_records()` (raise), и `validate_sources.py` (collect); пути raw/doc.md/state ВЫВОДЯТСЯ из папки (`raw_file`/`md_file`/`state_file`). `render_frontmatter()` порождает frontmatter. `validate_sources.py` — валидатор (структура + словари + уникальность id + relations + `relevance` + инварианты папок + для `geo_scope=national` форма `entity_id` == iso2, PR #5), возвращает `(errors, records)` — потребители (`run_pipeline`/`build_graph`) переиспользуют записи без повторного обхода дерева (PR #8); ненулевой код при ошибках, гейт также в CI (PR #5).
- **Словари** — `pipeline/vocab/vocab_*.yaml` (растут органически); `g2ai_pattern` = переносимый паттерн-решение (ось кросс-странового сравнения), `topics` = тема. `jurisdictions.yaml` (EU/ASEAN) даёт рёбра `member_of`. `axes` (`vocab_axes.yaml`, PR #26, `vocab-axes`) — аналитические оси триажа (`agentic_g2ai`/`digital_sovereignty`), растёт РЕДКО (в отличие от органических topics/g2ai_pattern — новая ось = пересмотр методологии): `Relevance.axis` вынесен из захардкоженного enum `Axis` (`schema.py`) в этот словарь, гейт — `validate_sources.py`, консистентно с doc_type/authority/topics/g2ai_pattern. Правило проекта: enum — если код ветвится по значению ИЛИ это упорядоченная шкала (`TargetFit`/`AssessedStage`/…); словарь — если номинальная содержательная классификация.
- **Граф** — `build_graph.py`: гетерогенный NetworkX-граф (узлы document/pattern/topic/issuer/jurisdiction; рёбра exemplifies/about/published_by/applies_to/member_of + document→document из relations). Экспорт GraphML. Запросы: `docs_by_pattern`, `docs_in_bloc`, `lineage`.
- **Поиск** — единый `pipeline/index/corpus.db` (SQLite): `corpus_index.py` (канонические ~512-токенные чанки + FTS5 `unicode61`; `build`/`search`), `vector_store.py` (векторы + брутфорс-косинус; `embed-corpus`/`vsearch`). Чанки канонические — общие для FTS и векторов. **Инкрементальность двухуровневая (PR #9):** документ — единица переиндексации FTS (`doc_state(doc_id, fingerprint)`: пере-чанкуются и пере-индексируются в FTS через external-content `'delete'`/`'insert'` ТОЛЬКО документы с изменившимся `doc.md`; `--force`/смена `chunk_max_tokens` → эталонный полный rebuild; инкремент — одна транзакция, `doc_state` не обгоняет chunks); чанк — единица эмбеддинга, ключ **`content_hash`** (sha256 `breadcrumb + text`, не эфемерный `chunk_id`; breadcrumb — с PR #15, см. ниже): эмбеддятся только новые хэши + `gc_vectors` осиротевших, общий boilerplate-чанк = один вектор на всех носителей. Класс дефекта «вектор указывает на чужой текст» устранён ПО ПОСТРОЕНИЮ → стоп-гэп `DROP vectors`/`vectors_fingerprint`/`VectorsStaleError` (PR #7) упразднён, заменён честным missing-репортом vsearch. `index_meta` (key-value) хранит `corpus_fingerprint` (stat-based глобальный гейт «есть ли работа», реконсиляция пересборки; meta-aware с PR #15 — см. ниже), `chunk_max_tokens`, `schema_version` (v3 с PR #15; открытие легаси-БД пересоздаёт производные таблицы). `corpus_index.py search` по умолчанию экранирует запрос (`sanitize_fts_query` — дефис не входит в bareword-грамматику FTS5), `--raw` — для честного FTS5-синтаксиса.
- **Retrieval-фасад (PR #15, `analyze-retrieval`)** — `chunk_text()` секционирует текст по markdown-заголовкам ДО packing: у каждого чанка `breadcrumb` (цепочка заголовков-предков, `"H1 › H2"`), чанк никогда не пересекает границу секции; `embed_input(breadcrumb, text)` — что реально видит эмбеддер (не голый текст чанка). Схема v3: двухколоночный FTS (`text`, `breadcrumb`) + новые таблицы `doc_facets`/`topics_map` (наполняются из курируемых записей, полная перезапись каждый прогон индексации, дешевле diff-логики); `corpus_fingerprint` — meta-aware (стат и `doc.md`, и `meta.yaml`) — правка тира/топиков в `meta.yaml` без изменения текста триггерит обновление фасетов, но НЕ пере-чанковку/пере-эмбеддинг документа. Единая точка retrieval — `analyze/retrieve.py::retrieve()`: RRF-слияние (`RRF_K=60`/`POOL=50` — константы, не конфиг) FTS5+вектор с фасетными фильтрами (entity/topic/doc_type/authority/axis/target_fit → `doc_facets`/`topics_map`); `embedder=None` — честная FTS-only деградация (для CI/свежей БД без векторов). CLI `hsearch.py` — тонкий фасад над `retrieve()` (`--backend bge|openrouter|none`). `ab_eval.py` — режимы `--mode fts|vector|hybrid|all`; контрольные запросы вынесены в трекаемый `pipeline/config/eval_queries.yaml` (не в коде). **Эмпирическая находка (не дефект):** FTS-режим даёт ~0% hit@k на запросах-перефразировках (implicit-AND по всем токенам полного предложения почти никогда не матчится в естественной прозе) — ровно причина существования гибрида; кросс-язычный EN→cnr retrieval через векторный канал подтверждён рабочим на реальном документе (`me-crps-registration-law-2025`).
- **Эмбеддинги — API-first (PR #16, spec embed-api-first, 2026-07-18)** — `embed.py`: production-путь `OpenRouterEmbedder` (`DEFAULT_BACKEND="openrouter"`, `DEFAULT_CLOUD_MODEL="google/gemini-embedding-001"`, MRL-срез до `DEFAULT_CLOUD_DIMS=1024` НА КЛИЕНТЕ; неймспейс векторов — с суффиксом размерности `@1024`; `"dimensions"` в payload НЕ передаётся — провайдер с фиксированной размерностью ОТВЕРГАЕТ запрос, живой факт nemotron). Модель выбрана живым A/B (пользователь+Fable 5): hit@10=100% на en/cnr/et, чинит et-слабость bge (60%→100%); qwen3-8b (фаворит MMTEB) и nemotron:free провалили критерии — **опубликованным бенчмаркам не верить без локального A/B (второй подтверждённый случай после EmbeddingGemma)**. Операционная броня: retry/backoff `RETRY_SCHEDULE` (429/5xx/сеть/error-в-теле-200 через `_InbandError` → до 5 попыток; прочие 4xx — немедленный отказ с телом); чекпоинтинг `embed_and_store` батчами по 64 (обрыв ЛЮБОЙ природы теряет ≤1 батч, рестарт добирает по `not_embedded_for` — kill-9-тест пройден живьём); sensitivity-гейт `chunk_hashes(exclude_all_carriers_in=confidential_doc_ids(conn))` — чанк, ВСЕ носители которого confidential, облаком не эмбеддится (боилерплейт с public-носителем — эмбеддится), локальный бэкенд без фильтра; `doc_facets.sensitivity` — аддитивная колонка без бампа схемы (`_ensure_facets_sensitivity`). Локальный ФОЛБЭК — `OnnxBgeEmbedder` (bge-m3 int8; офлайн/confidential, НЕ основной путь). `ab_eval.py` — `--backends bge|openrouter:<model>` (comma-список) + per-language срезы (`lang` в `eval_queries.yaml`: en/cnr/et). Токенизатор ЧАНКОВКИ остаётся `bge_tokenizer.py` при любом эмбеддере; `.env` (`OPENROUTER_API_KEY`) — `env.py`.
- **Оркестратор** — `run_pipeline.py`: идемпотентный end-to-end по корню `sources/` (download→convert→frontmatter→index). Реконсиляция desired-state (работа выводится из ФС: наличие `raw.*` + sha256 из `.state.yaml`, свежесть `doc.md`, синхронность frontmatter) — повторный прогон no-op, самовосстановление; пересборка индекса реконсилируется тем же принципом — по `corpus_fingerprint` (глобальный гейт, не in-run флаг, PR #7), а сама переиндексация инкрементальна по `doc_state`/`content_hash` (PR #9): меняется только изменившийся документ, эмбеддятся только новые хэши. `--embed` реконсилирует векторы при пересборке FTS (бэкенд — `--embed-backend {openrouter,bge}`, дефолт облачный; отказ облака после ретраев НЕ рвёт FTS-часть — она закоммичена раньше, ошибка репортится с ненулевым exit); standalone-добор при актуальном FTS — `vector_store.py embed-corpus`; сводка прогона печатает счётчик чанков без векторов production-неймспейса с подсказкой добора (PR #25 — дефолтный прогон сознательно не эмбеддит, но об отставании векторного слоя больше не молчит). Ручной/неотслеженный raw (`--no-download`) усыновляется — sha256 считается и фиксируется реконсиляционным шагом, не только при автоскачивании (PR #7). Атомарная запись артефактов документа — через `fsio.py` (единая staging-политика, dot-префиксные файлы вне глобов `raw.*`, PR #6/#8). Отказ документа не рвёт батч (включая стадию планирования, PR #6); quality-gate (валидация реестра), `--dry-run`/`--force`/`--only`/`--no-download`/`--embed`/`--watch-dir`. Сознательно БЕЗ Airflow/Prefect/Dagster (серверные, оверкилл для масштаба/железа).
- **Добыча / WAF-устойчивость** — `acquisition.py` (спек `docs/pipeline/acquire/tech_specs/source-acquisition-ladder/spec.md`, PR #2): лестница `direct → official_alt → manual → archive` на стадии download. Детекция блока — не только HTTP-код, но и Cloudflare-отпечатки (`cf-ray`/`__cf_bm`/challenge-маркеры в теле) и content-magic, иначе `200`-challenge сохранился бы как «документ» (реальный кейс: ai.gov.ae, error 1020, заблокировал и `curl`, и автоматизированный браузер); сетевые отказы curl (DNS/connect, exit 6/7) классифицируются как `dead` и уходят в archive-ступень, а не роняют документ (PR #6). `manual` — 1-клик путь (`xdg-open` официального URL + watch-folder поверх системной папки загрузок, `xdg-user-dir DOWNLOAD`), синхронен только при `--only` (батч на нём не виснет, просто репортит и идёт дальше); кандидаты сверяются с title документа (грубый порог по словам, с мемоизацией парсинга) — посторонний PDF в папке загрузок больше не может тихо стать оригиналом; явный `--watch-dir` отключает сверку (PR #6). `archive` — снимок из Wayback CDX API для подтверждённо мёртвых URL (`fidelity: archived_snapshot`, никогда не `live`; `sensitivity: confidential` отключает archive — Wayback публикует обращение; запрос берёт РЕАЛЬНО свежайший снимок через `limit=-N`, не первые N по возрастанию, PR #6). Состояние добычи (`acquisition_method`/`fidelity`/`acquisition_checked`/`retrieved_snapshot_date` + sha256) пишется в `.state.yaml`-sidecar документа (`schema.save_state`, plain YAML — файл машиннописаный, без комментариев; `ruamel`/round-trip и `persist_acquisition_state` убраны как ненужные после разделения курируемого/операционного — corpus-layout-v2, PR #4).

- **Триаж релевантности / слой кандидатов** — `schema.py` + `validate_sources.py` (спек `docs/pipeline/acquire/tech_specs/source-relevance-triage/spec.md`, PR #3): гейт-ФИЛЬТР допуска источника в реестр, перед добычей. Кандидаты — в **отдельном `sources/candidates.yaml`** как `CandidateRecord` (лёгкий `extra="allow"` тип с провенансом добычи; НЕ `status:candidate` на `SourceRecord`); допуск = конверсия типа `promote_candidate() → SourceRecord`, не смена статуса. Релевантность двухстадийна (метаданные+WebSearch ДО добычи → подтверждение по тексту ПОСЛЕ конвертации), по двум осям (`agentic_g2ai` узкая / `digital_sovereignty` широкая), тир `primary`/`context`/`background`. Дата — сигнал, не гейт: `frontier_year` в `pipeline/config/triage.yaml`. Провенанс добычи остаётся в `candidates.yaml` (в `meta.yaml` НЕ копируется — corpus-layout-v2). Автоматический Стадия-1-судья — отложен (пока триаж ручной). **Генератор кандидатов (DISCOVERY)** — чартер `docs/pipeline/discovery/charters/architecture.md` (мультиконнекторный портфель: registry/outlet_watcher/directed_search/manual). **Ядро реализовано** (`discovery-core`, PR #23, 2026-07-21): подпакет `discovery/` — протокол `Connector` (коннектор — чистая функция от курсора, не пишет в store сам), реестр по паттерну `_CONVERTERS`, кросс-коннекторный dedup (`normalize_url`/`normalized_title`, юникод-aware, без fuzzy-библиотек — детерминизм; ключи по убыванию надёжности: URL → issuer+title+date → content_hash; отклонённые триажем кандидаты не ре-инжектятся), атомарный персист `sources/candidates.yaml` + курсоры `sources/.discovery_cursors.yaml`, оркестратор с изоляцией отказов (зеркало `run_pipeline.process_docs`) и идемпотентной реконсиляцией. CLI `discover.py discover [--only id ...] [--dry-run]`. **Ручной/directed-каналы реализованы** (`discovery-manual`, PR #25, 2026-07-21; живой смок на GAIRI 2025 → первый документ трека research-papers): `inject` (типизированный завод кандидата через dedup — повторный inject той же ссылки no-op, отклонённые не воскресают), `worksheet` (таблица ждущих РЕКОНСИЛЯЦИЕЙ против реестра, без хранимого статуса; шапка = самодостаточный формат решений), `apply` (батч promote/reject по `raw_hash`-ключу, изоляция отказов per-решение, `--dry-run`; после не-dry-run батча с ≥1 admit сам зовёт `validate_sources` и печатает расхождения словаря с ненулевым exit — PR #26, гейт постфактум: `meta.yaml` уже записан к этому моменту, опечатка в axis/doc_type/authority видна сразу в выводе apply, а не только следующим отдельным прогоном) + скилл `.claude/skills/directed-search/{SKILL.md,README.md}` — протокол воспроизводимых поисковых кампаний (сетка запросов термин×юрисдикция×язык, верификация издателя по метаданным, inject — единственная дверь; ПЕРВЫЙ трекаемый скилл: методология провенанса корпуса, класс triage.yaml). **Упакован как Agent Skill** (`agent-skills-packaging`, PR #27, 2026-07-21): перенесён из Claude Code-специфичной слэш-команды (`.claude/commands/`) в открытый стандарт SKILL.md — читается Claude Code/Agent SDK/Copilot и др. без правки кода; README даёт три способа запуска (Claude Code / Agent SDK-рецепт / любая OpenRouter-модель через Crush). Словарные/константные значения внутри протокола (ось, лимит `--summary`) указывают на источник истины (`vocab_axes.yaml`, `schema.CANDIDATE_SUMMARY_MAX`), а не дублируют его инлайном — та же дисциплина, что и в схеме. **Слим `CandidateRecord` (решения куратора на ревью PR #25):** `connector_kind`/`source_ref`/`reported_status` убраны (архетип канала = грамматика `connector_id`: `manual` | `search:<кампания>` | id кодового коннектора; enum ConnectorKind остался в протоколе Connector/API inject), `native_summary` ≤600 знаков (жёсткий отказ), `native_tags`/`matched_vocab_tags` дефолт None; дамп: `title` первым, пустая строка между записями. **Конвенции decisions.yaml:** `id` первым / `action` последним (код к порядку безразличен), содержательный контент EN, `rationale` = ТОЛЬКО факторы релевантности (не пересказ summary); файл эфемерен (один заход триажа). **Дефолты admit:** `authority` — из doc_type, `track` — из jurisdiction/issuer_type (явный ключ побеждает — кейс draft; эхо «по дефолту: …» в сводке; аннотация `hidden_fields`); дефолты материализуются в meta.yaml полными значениями — слой знаний ничего не теряет. Первый registry-коннектор (ETO AGORA, `discovery-agora`) — позже.

Модель bge-m3 (~543 МБ) качается отдельно в `pipeline/models/bge-m3-onnx-int8/` (int8 ONNX, команды — в проектном плане). Тесты, зависящие от модели, помечены `@pytest.mark.model` и в CI пропускаются (`-m "not model"`). Векторы разных моделей несравнимы — на корпусе живёт одна модель за раз.

## Слой конвертации: реестр форматов, графика, HTML, OCR, облачный VLM-тир, DOCX
Мультиформатный `raw.* → doc.md` слой (чартер `docs/pipeline/convert/charters/architecture.md`; серия спеков `docs/pipeline/convert/tech_specs/` — все реализованы). Оркестратор (`_do_convert`) форматов не знает — `converters.resolve_converter(raw)` диспетчеризует по расширению.

- **Реестр** (`convert/converters.py`, PR #11) — `_CONVERTERS: dict[str, Converter]` (ключ — расширение `raw.*`), `Converter(name, version, convert)`; смена `version` → авто-реконверсия при следующем прогоне (`needed_stages` сверяет `converter_name/version` с `.state.yaml`). Typed-отказы: `UnsupportedFormat` (нет конвертера под расширение), `NeedsOCR` (PDF без текст-слоя — дешёвый проход `_detect_scan` ДО полной конвертации, `SCAN_MIN_TEXTPAGE_FRACTION=0.5`), `ConversionError` (база). Golden-самосверка (`test_convert_golden.py`, `@pytest.mark.corpus`): свежая конвертация каждого локального `raw.*` должна байт-в-байт совпасть с уже лежащим `doc.md` — эталон корпус сам себе (тела документов не в git).
- **Графика/инфографика** (`convert/pdf_graphics.py`, PR #12) — интегрирован в `pdf_to_markdown.py`, только цифровой PDF-путь. Заменил старую word-gap-эвристику (см. историческую сноску ниже) на векторную геометрию: rects/curves/images страницы кластеризуются, три guard'а (мин. элементов, макс. доля страницы, макс. текста-на-элемент — «сомнение → проза») отсеивают декор/боксированную прозу от настоящих фигур; грид→таблица, одноосевая последовательность→список, растр — дедуп по content-хэшу (декор молча, `RASTER_DECOR_MIN_REPEATS=3` повторов по документу) + маркер крупных уникальных (`RASTER_MIN_PAGE_FRACTION=0.08` площади страницы, `> [Image, p. N — raster content not analyzed]` — текст не теряется, честно помечена неопределённость). Bbox-anchored позиционная вставка (фикс «все таблицы/картинки съезжали в конец документа»). `pdf`-конвертер → v2.
- **HTML** (`convert/converters.py::_convert_html` + `convert/eli.py`, PR #13) — `trafilatura==2.1.0` (markdown, `favor_recall=True`). `SourceFormat` (`pdf`/`html`/`docx`) в `SourceRecord` — куратор объявляет формат при триаже, диктует расширение `raw.*`, ветку `acquisition.classify_response(expected=)` (WAF-челлендж-детект ДО content-type — челлендж тоже `200 text/html`) и `find_wayback_snapshot(mimetype=)`. `eli.py` — ELI-разметка (EUR-Lex/CELLAR Publications Office ЕС: `CHAPTER`/`SECTION`/`Article`/`ANNEX` как id-префиксные `<div>`, НЕ семантические `<hN>`) промоутится в настоящие `<hN>` до `trafilatura` + оборачивает `<body>` в `<article>` (без этого trafilatura ТЕРЯЕТ все заголовки при выборе «главного контента» — эмпирически подтверждено трассировкой; риск общий для любого документа без явного `<article>`/распознаваемых class/id, не специфичен для EU AI Act).
- **OCR** (`convert/converters.py::_ocr_normalize` + `convert/ocr_headings.py`, PR #14) — сканы без текст-слоя: `ocrmypdf --skip-text` (subprocess, системный apt, языки по `TESSERACT_LANGS` — `cnr/sr→srp_latn`, мультиязычный резерв ru/ar/zh/es/ja/de/fr/it, CJK/арабский без `+eng`-суффикса) кладёт невидимый текст-слой **IN-PLACE в `raw.pdf`** — один PDF-файл на документ, не отдельный сайдкар; `.state.yaml` пересчитывает sha256/размер/mtime сразу после конвертации (иначе следующий stat-guard принял бы мутацию текст-слоя за порчу файла и затребовал бы передобычу). Т.к. raw «вылечивается» от статуса скана навсегда, повторное распознавание «это OCR-документ» на будущих переконвертациях (`--force`/бамп версии конвертера) — по метаданным `ocrmypdf` (`Creator`-поле PDF, `_was_ocr_normalized()`), переживающим саму мутацию текст-слоя. `ocr_headings.py` — отдельный markdown→markdown пост-проход, precision-first, три тира по убыванию точности (ключевые слова ANNEX/CHAPTER/SECTION/Article + региональные ČLAN/CLAN/GLAVA/PRILOG/ODJELJAK/DIO → CAPS-фраза+тело → голая нумерация ≤2 уровней): штатная кластеризация `pdf_to_markdown.py` по размеру шрифта калибрована на чистых кеглях цифровых PDF и на OCR-bbox (геометрический шум OCR, не типографика) даёт ненадёжные/противоречивые уровни, поэтому на OCR-ветке она полностью ИГНОРИРУЕТСЯ — `ocr_headings` снимает любую её разметку и переоценивает строку заново сама (единый источник истины, не два несогласованных). Живой полевой тест (`me-crps-registration-law-2025`, ME, первый скан в корпусе) — известные системные ограничения, найденные им, задокументированы в самом спеке (`docs/pipeline/convert/tech_specs/convert-ocr/spec.md`, раздел «Известные ограничения после живого теста»); один класс (OCR-искажение текста — спутанные диакритика/цифры/римские числа) сознательно НЕ чинится — попытка угадывать/подправлять текст опаснее для юридического корпуса, чем косметическая неточность в заголовке.
- **Hardening + good-enough заморозка** (`convert-hardening`, разовый пакет 2026-07-19) — A1: bold-фолбэк заголовков (`pdf_to_markdown._bold_heading_level`) — документ с одинаковым кеглем заголовка/тела больше не пропускается целиком: строка целиком bold, кегля не мельче тела, короткая, без хвостовой пунктуации/маркера списка промоутится в заголовок УРОВНЕМ НИЖЕ всех размерных уровней документа. A2/A3: многостраничные таблицы, фрагментированные с повтором шапок и пустыми колонками (Эстония) — `merge_split_tables` (markdown-пост-проход, сливает только соседние блоки с ИДЕНТИЧНОЙ шапкой) + `_drop_empty_columns`; `pdf`-конвертер → v4. B1/B2: `convert/html_preprocess.py` — first-match Strategy-реестр HTML-препроцессоров (следующий legal-портал регистрируется одной строкой, не правкой `_convert_html`; ELI — первая запись); guard потери заголовков (`<hN>` в исходнике, но ни одного `#` в выходе → `logger.warning`, не отказ). C1: `convert/lint.py` — авто-QA вместо ручного аудита каждого документа (`no-headings`/`text-loss: <ratio>`/`table-ragged: <n>`), пишется в новое поле `OperationalState.lint_defects` (`.state.yaml`), НИКОГДА не роняет конвертацию. C2 — политика good-enough: после пакета калибровка PDF/HTML-конвертеров ЗАМОРОЖЕНА — правка только при (а) дефект блокирует анализ Стадии 2/evidence-пак, (б) накопился класс ≥3 однотипных дефектов, (в) новый формат; альтернатива локальной перекалибровке — точечная облачная эскалация документа (реализовано `convert-cloud-tier`, см. ниже). Полный построчный аудит — только первый документ НОВОГО конвертер-пути; далее — только флаги C1 + выборочный взгляд. Отложенные точечные доработки (OCR §3.2–3.4 `convert-ocr` и любая иная пер-документная калибровка) — заморожены до срабатывания триггера (б)/(в).
- **Облачный VLM-тир** (`convert-cloud-tier`, PR #18) — точечная эскалация в облако вместо/поверх локальной калибровки, по триггеру C2 выше. `core/openrouter.py` — общий chat/completions-клиент (`RETRY_SCHEDULE`, `InbandError` — та же броня, что у `embed.py`). `convert/cloud_ocr.py` — постраничный VLM-OCR сканов (рендер `pdfplumber`→JPEG, батчи по `OCR_BATCH_PAGES=20`/`OCR_BATCH_MAX_MB=8`, кэш-сайдкар `.cloudocr.parts.yaml`, `DEFAULT_VLM_MODEL="google/gemini-3-flash-preview"` — победитель живого A/B на OCR И фигурах); интегрирован в `_convert_pdf` как альтернатива tesseract-пути. `convert/lint.py::witness_checks` — witness-гейт: облачный вывод сверяется со свидетелем (локальный tesseract-текст) по словарному token-recall и числовому инварианту (мультимножество числовых токенов) — расхождение пишется в `lint_defects`, не блокирует конвейер (самый опасный класс ошибок VLM — тихая подмена цифры при внешне безупречном тексте). `convert/figures_vlm.py` — VLM-семантика фигур (реализация чартера §4.3): скан doc.md на маркеры → рендер региона → VLM → кэш `.figures.yaml` (см. ниже про DOCX-объекты — общий кэш-механизм). `run_pipeline.py`: стадия `figures` реконсилируется по `has_bare_markers` (ФС-based desired-state, как остальные стадии), флаги `--no-cloud`/`--vlm-model`. `ocr_headings.py` получил additive-режим для облачного вывода (не выбрасывает разметку, как для tesseract-ветки, а дополняет). Живой чекпоинт: `me-crps-registration-law-2025` (скан) + `me-undp-aila-2025` (UNDP AILA, cnr) — ноль галлюцинаций, числа сошлись.
- **DOCX** (`convert-docx`, PR #20, 7 коммитов) — третий `SourceFormat`. `convert/docx_groups.py`: XML-препроцессор ДО mammoth — детектирует composite-группы фигур (`mc:Choice`▸`wpg:wgp`) и голые нативные чарты (`w:drawing`▸`c:chart` вне AlternateContent, данные в `embeddings/*.xlsx`) в `word/document.xml`, вырезает объект целиком, заменяет текстовым сентинелом (`id12 = sha256(XML-поддерево)[:12]`) — сентинел проходит текстовый путь как обычный текст и сохраняет ТОЧНУЮ позицию фигуры в потоке; captions собираются из поддерева (для группы) или `c:title` чарт-парта (для чарта) ДО вырезки — zero-loss даже без VLM. chartEx (sunburst и т.п.) сознательно НЕ детектируется отдельно — Word кладёт готовую fallback-PNG в `mc:Fallback`, её забирает штатный mammoth-путь. Текстовый путь — прямой `mammoth` (custom image-handler, кастомный `MarkdownConverter`-сабкласс `markdownify`, НЕ `markitdown` — версийный скос приватного API), фолбэк-листинг `word/media/*` с **orphan-фильтром** (маркер только для media, реально референсированных XML-частями — посторонние файлы в архиве не всплывают даже маркером, privacy по построению). `figures_vlm.py` получил 4-ю грамматику маркера (`docx group|chart`) и рендерер `_render_docx_group`: изолированный мини-docx (полный zip, body = один блок) → `soffice --headless` → PDF → кроп по bbox контента (клэмп к границе страницы) → JPEG → тот же VLM-вызов и кэш `.figures.yaml`, что у PDF-фигур. Требует системный LibreOffice (`soffice`) для рендера группы/чарта — отсутствие/отказ деградирует до маркера+captions, конвертация не падает (маркер pytest `libreoffice`). Живой стресс-тест (8 разнотипных объектов: 5 wpg-групп, нативный чарт, chartEx-sunburst, таблица 28×9) — ноль фабрикаций в VLM-выводах, все числа графика точны.
- **XLSX** (`convert-xlsx`, PR #22) — четвёртый `SourceFormat`. Безпотоковая (cell-anchored) модель: у `.xlsx` нет линейного потока прозы, документ — сетка листов/ячеек. `converters.py::_convert_xlsx` — многолистовая GFM-таблица на лист (`openpyxl`, НЕ `calamine` — нужен доступ к merged-cell структуре и chart-XML-частям), слитые ячейки схлопываются в якорную, скрытые листы включаются с маркером `(hidden)` (честность важнее приватности), формулы — только закэшированное `data_only=True`-значение. `convert/xlsx_charts.py` — детект встроенных чартов (`xl/charts/*.xml` через `xdr:from`-якорь), `id12` по XML-структуре чарта (тот же принцип, что у docx-групп). `figures_vlm.py`: 5-я грамматика маркера + `extract_chart_workbook` — изолированная мини-книга под рендер. Живой чекпоинт (внешний стресс-тестовый World Bank GovTech-документ, 9 листов/55 чартов, **НЕ в корпусе** — решение пользователя, только материал для теста; вырезка листа `Stats` осталась как gitignored фикстура `tests/fixtures/local/`) опроверг наивный дизайн («скрыть остальные листы») — soffice игнорирует `sheet_state` при headless-экспорте, листы теперь физически удаляются + формулы девалидируются до закэшированных значений (иначе `#ИМЯ?` на удалённые листы глушит даже `c:numCache` самого чарта) + форсированный `_xlnm.Print_Area`/`fitToPage` на одну страницу. При нескольких чартах на одном листе — mutual ownership система (`_ownership_ranges`/`_blank_foreign_cells`, по духу как взаимное исключение docx-групп, но для общей сетки ячеек: `data_keep` собственных `<c:f>`-диапазонов ВСЕГДА побеждает чужое `exclude`, визуальный `bbox_keep`-запас — нет), `_MAX_TABLE_SPAN=25` не даёт полноценному датасету раздувать печатную область, асимметричный запас bbox (`col=1`/`row=2`, найден экспериментально). Общий `FIG_PROMPT` (эффект кросс-форматный — влияет и на PDF/docx) доработан по итогам чекпоинта: mermaid-триггер расширен с flowchart-only до `pie`/`xychart-beta`/`radar-beta` (сверено live через реальный `mermaid.parse()`) + `classDef`/`style` для цветовой группировки flowchart-узлов (нейтральные имена классов, НЕ danger/success — не подходит корпусу); явный запрет транскрибировать "родную" таблицу-источник чарта отдельно от самого чарта + молчаливое CROSS-CHECK числовой сверки. Кросс-форматный фикс-сателлит: row-aware разрезка оверсайз-GFM-таблиц в чанковке (`index/chunking.py`, PR #21, сделан перед этим спеком — xlsx-таблицы обычно крупнее PDF/DOCX-аналогов).

## `pipeline/scripts/convert/pdf_to_markdown.py` — как устроен и его реальные ограничения
Полностью автоматический конвертер PDF→Markdown на `pdfplumber` (MIT, pure Python, CPU-only, ~70MB venv — сознательно НЕ marker-pdf/datalab-to, см. ниже почему). Механизмы:
- **Восстановление порядка чтения в многоколоночной вёрстке** — geometric gap-анализ по bounding-box словам (`detect_columns`: проекция покрытия по x, поиск устойчивого по высоте разрыва в центральной зоне страницы), а не regex-эвристика поверх готового текста.
- **Иерархия заголовков** — по кластерам font-size конкретного документа (пересчитывается заново для каждого PDF из его собственной гистограммы), НЕ по regex на нумерации "1.1.1" — работает и для документов без явной нумерации разделов. Только цифровой PDF-путь — на OCR-ветке эта кластеризация ненадёжна (шум bbox, не типографика) и полностью игнорируется в пользу отдельного `ocr_headings.py` (см. «Слой конвертации» выше).
- **Колонтитулы/номера страниц** — частотный анализ ЦЕЛЫХ строк (не отдельных слов) в верхней/нижней полосе страницы.
- **Сноски** — отдельный (заниженный, ~65% от размера тела) порог только для крошечных надстрочных номеров-маркеров — сам текст сноски обычно почти неотличим по размеру от текста обычных врезок/кейс-стади, попытка развести их одним порогом теряла бы кейс-стади.
- **Таблицы** — `pdfplumber.find_tables()` + фильтр по АБСОЛЮТНОМУ числу непустых ячеек (не по доле — у настоящих таблиц со слитыми ячейками доля заполненности так же низка, как у шумных обломков диаграмм).
- **Диаграммы/инфографика** — векторная геометрия через `pdf_graphics.py` (PR #12, детали — «Слой конвертации» выше); прежняя word-gap-эвристика по тексту заменена полностью (её слабости — см. абзац «Известное ограничение» ниже).

**Известное ограничение — пороги изначально калибровались на одном документе** (Сингапур, IMDA Model AI Governance Framework for Agentic AI v1.5, 2026-07-15) и не переносились вслепую на следующие страны. *2-й прогон (Эстония, MKM «Tehisintellekti tegevuskava 2024–2026», эстонский, 47 стр., 2026-07-15) ПОДТВЕРДИЛ слабые места ИСХОДНОЙ word-gap-эвристики диаграмм (ложные срабатывания на оглавлении/SWOT-матрицах) — с тех пор заменена векторной геометрией (`pdf_graphics.py`, PR #12, см. «Слой конвертации»); многостраничные таблицы фрагментировались с повтором шапок и пустыми колонками — закрыто `convert-hardening` A2/A3 (`merge_split_tables`/`_drop_empty_columns`, 2026-07-19, см. «Слой конвертации»). Извлечение текста, заголовков и диакритики (õ/ä/ö/ü) — чисто.* Актуальная конкретика порогов:
- `MIN_GAP_PT=14.0`, `COLUMN_ZONE=(0.30, 0.70)`, `MIN_GAP_HEIGHT_FRAC=0.45` (детектор колонок) и `TINY_MARKER_RATIO=0.65` (сноски) — абсолютные числа, не относительные к кеглю конкретного документа
- Детектор колонок ловит только ОДИН разрыв (максимум 2 колонки) — 3-колоночная вёрстка не поддержана
- Заголовки различаются в основном по РАЗМЕРУ шрифта; документ с одинаковым кеглем заголовка/тела закрывает bold-фолбэк A1 (`convert-hardening`, 2026-07-19) — строка целиком bold, кегля не мельче тела, короткая, без хвостовой пунктуации промоутится в заголовок НИЖЕ всех размерных уровней; заголовки без ЛЮБОГО типографского выделения (ни размером, ни насыщенностью) остаются слепым пятном (цифровой путь; для OCR-пути см. `ocr_headings.py` — тоже своя, отдельно задокументированная граница возможностей)
- **Специфический риск для Японии** (в seed-списке `sources/temporary/small_states_v01.md` есть японская стратегия): `pdfplumber.extract_words()` токенизирует по пробелам — если исходный PDF на японском (CJK, без пробелов между словами), группировка в строки/колонки может не сработать; проверить наличие англоязычной версии до прогона.

**Практика на каждый новый документ:** тот же выборочный аудит, что делался для Сингапура — сверка списка заголовков, проверка таблиц на дубли/фрагментацию, чтение хотя бы одного «сложного» (многоколоночного/с диаграммой) участка целиком. Не доверять результату вслепую.

## Почему НЕ marker-pdf / НЕ Jina Reader (сравнение проведено 2026-07-15)
- **marker-pdf** (datalab-to) — GPL код + Open RAIL-M веса, тянет ~1.3GB+ CUDA-стека (torch/cudnn/nccl) даже на CPU-only задаче. На рабочей машине физически несовместим: GeForce 940MX (Maxwell, compute capability 5.2) выпала из поддержки CUDA 13.x/актуального PyTorch — установку пришлось прервать и полностью вычистить.
- **Jina Reader** (r.jina.ai) — по умолчанию PDF.js (не LLM), но при живом тесте на реальном гос. PDF (тот же сингапурский документ) получил 403 Forbidden — WAF издателя (tech.gov.sg) заблокировал IP Jina, хотя прямой `curl` с той же машины скачал файл без проблем. Системный риск, специфичный именно для корпуса государственных документов — у многих гос. сайтов WAF банит дата-центровые/прокси IP.
- `pdfplumber` — ноль GPU-зависимостей, работает локально, не проксирует запросы через сторонние серверы (важно, т.к. документы могут быть чувствительны к раннему разглашению до публикации итогового пакета предложений).

## Известные факты о рабочей машине
Linux Mint 22.3 Cinnamon, ноутбук 2015 года (Intel Core i5-6200U, 2 ядра/4 потока), 8GB RAM, дискретная GPU (GeForce 940MX, Maxwell) НЕСОВМЕСТИМА с современным CUDA/PyTorch — см. предупреждение выше про marker-pdf. Интернет — LTE-роутер, ~1MB/s, сельская местность. Перед предложением любых GPU-зависимых ML-инструментов — сначала проверять CPU-fallback реалистичность или явно предупреждать перед многогигабайтной загрузкой.

**(Ре-)эмбеддинг корпуса — облачный по умолчанию (PR #16, 2026-07-18): минуты, не часы.** Живой замер на смок-приёмке: полный корпус (4 док, 628 чанков ≈ 322k ткн) — ~1–1.5 мин на модель (gemini-001@1024, LTE). Целевые 1000 док ≈ 80M ткн, wall-clock — часы СЕТИ, не CPU (машина свободна; экстраполяция по замеру ~322k ткн/мин ≈ 4–5 ч последовательно). Обрыв любой природы (сеть/kill/OOM/закрытие VS Code) теряет ≤64 чанка — чекпоинтинг `embed_and_store`, рестарт добирает инкрементально (проверено kill-9-тестом). ⚠ OpenRouter-кредиты: перед батч-наполнением сверять баланс с прогнозом расхода (актуальные прогнозы/замеры — во внутренних записях). При ЛОКАЛЬНОМ фолбэке (`--embed-backend bge`) прежние правила в силе: длительность определяется ЧИСЛОМ ЧАНКОВ и давлением на 8GB RAM (живой инцидент PR #15: свободно ~200 МБ, своп 1.7–3.5 ГБ — прогон кратно дольше ориентира) — перед оценкой проверять `free -h`/`vmstat`.

**Политика (решение пользователя 2026-07-19): бюджет и стоимость работы пайплайна — не для показа наружу.** Денежные суммы, тарифы, балансы и оценки расходов НЕ попадают ни в какие коммитящиеся артефакты: трекаемые файлы (включая этот), сообщения коммитов, тела PR, docstrings/комментарии кода. Их место — только внутренние, gitignored записи: memory ассистента и `docs/` (спеки/бэклог).

**Фоновые процессы не гарантированно переживают закрытие VS Code/терминала — даже будучи POSIX-detached.** Эмпирически подтверждено (PR #15): длинный фоновый прогон (`vector_store.py embed-corpus`/`ab_eval.py`), запущенный через harness `run_in_background`, жил в СОБСТВЕННОЙ POSIX-сессии (свой `SID`, без управляющего терминала — проверено `ps`/`/proc`), тем не менее был убит при закрытии/перезапуске VS Code (дважды, потеряно ~30-40 минут вычислений каждый раз). Точный механизм не установлен — НЕ `systemd-logind` (`KillUserProcesses=no`, залогиненная сессия не завершалась) — вероятно cgroup-очистка терминала VS Code. **Не полагаться** на выживание длинных фоновых задач при закрытии хост-приложений. Для эмбеддинга риск ЗАКРЫТ чекпоинтингом (PR #16: `embed_and_store` батчами — смерть процесса любой природы теряет ≤64 чанка, рестарт добирает; бэклог §11); systemd user-service остаётся деплой-заметкой, не обязательным механизмом. Для ДРУГИХ будущих длинных фоновых задач принцип тот же: сначала чекпоинтинг, потом способ демонизации.
