# G2AI_ME — Claude Code memory

## О проекте
Независимый исследовательский трек. Две параллельные, но связанные линии работы:

1. **G2AI-корпус** (`sources/`, `pipeline/scripts/`) — сбор и структурирование международного опыта государств по построению политики/инфраструктуры/регулирования под GenAI и агентные сервисы («государство как MCP-сервер»), в фокусе малые страны, но не исключительно. Конечная цель — пакет предложений от независимого эксперта/отраслевой ассоциации, включая черновик Стратегии ИИ Черногории до 2030, пока правительство её разрабатывает.
2. **Более широкое видение цифрового суверенитета Черногории** (`agenda/vision_and_plan.md`, v03) — параллельный (не строго вложенный в G2AI) трек: ядро локали `cnr`, мобильные клавиатуры, Wikipedia/Wikidata grounding, B2B2G с телеком-группами, дальняя цель — партнёрство государства с фронтир-лабой по «исландскому пути». G2AI-корпус — один из аналитических входов для этого видения, но не единственный и не главный по объёму документа.

Обе линии объединяет забота о точной и полной представленности Черногории в цифровом/ИИ-пространстве.

## Структура репозитория
```
sources/{track}/{entity}/{doc-id}/      # ПАПКА-НА-ДОКУМЕНТ (corpus-layout-v2): track=intl-xperience|montenegro; entity=слаг (sg/ee/oecd…)
#   ├─ meta.yaml    — курируемая запись (Dublin Core + topics/g2ai_pattern/summary/relevance); ТРЕКАЕТСЯ в git
#   └─ raw.pdf / doc.md / .state.yaml — оригинал / конвертация / операц.состояние (sha256/acquisition); НЕ в git
sources/candidates.yaml             # слой кандидатов (CandidateRecord) ДО допуска триажем (НЕ в git)
sources/temporary/small_states_v01.md  # seed-список (сырой вход discovery-manual, не источник истины; локально, НЕ трекается — под deny-default /sources/**)
pipeline/scripts/*.py               # код пайплайна (см. «Слой знаний» ниже): pdf_to_markdown (PDF→MD); schema/validate_sources (метаданные); build_graph (граф); chunking/corpus_index (FTS5); embed/vector_store/bge_tokenizer (семантика); ab_eval (A/B); acquisition (WAF-лестница добычи); run_pipeline (оркестратор); fsio (единая staging/atomic-write политика, PR #6/#8); env
pipeline/vocab/*.yaml               # контролируемые словари: doc_type/authority/topics/g2ai_pattern + jurisdictions (членство в блоках)
pipeline/config/triage.yaml         # тюнинг-параметры триажа релевантности (frontier_year); ТРЕКАЕТСЯ, правится вручную
requirements.txt, requirements-dev.txt, pyproject.toml  # зависимости (runtime/dev) и конфиг инструментов — в КОРНЕ репо
.github/workflows/ci.yml            # CI: ruff/mypy/pytest/validate_sources (модель-тесты @model в CI пропускаются)
pipeline/models/  pipeline/index/   # локальные, НЕ в git: скачанная модель bge-m3 / производный SQLite-индекс корпуса
agenda/vision_and_plan.md  # широкое видение цифрового суверенитета (см. выше; НЕ в git)
```
**`.gitignore` под `/sources/` — deny-by-default:** игнорируется всё (документы `raw.*`/`doc.md`/`.state.yaml`, `candidates.yaml`), КРОМЕ `meta.yaml` — курируемая мета (библиография/аналитика, не тела документов) версионируется в git, а сами документы (потенциально авторское право, см. «Статус» ниже) остаются локально. `agenda/vision_and_plan.md` и `pipeline/setup/` — тоже gitignored.

`.venv/` в КОРНЕ репо — локальный venv через `uv` (в `.gitignore`, не коммитится; venv непереносим — не копировать, а пересоздавать: `uv venv .venv && uv pip install -r requirements-dev.txt`; классический `source .venv/bin/activate` у uv-venv может не работать — вызывать бинарники как `.venv/bin/…`). Конфиг инструментов — `pyproject.toml` (mypy/ruff/pytest; `pythonpath`/`mypy_path` указывают на `pipeline/scripts`, поэтому проверки запускаются ИЗ КОРНЯ, кеши тоже оседают в корне). Проверки перед коммитом изменений в `pipeline/scripts/*.py` (из корня, те же, что в CI): `.venv/bin/ruff check pipeline/scripts && .venv/bin/python -m mypy pipeline/scripts/*.py && .venv/bin/python -m pytest && .venv/bin/python pipeline/scripts/validate_sources.py` (в CI — `pytest -m "not model"`, т.к. модели там нет; `validate_sources.py` — 4-й шаг CI с PR #5).

## Пайплайн сбора документов
1. **Discovery + триаж релевантности (ДО добычи)** — WebSearch: найти реальный документ/издателя/дату (seed-список `small_states_v01.md` НЕнадёжен — реальные кейсы: фиктивные заголовки ОАЭ и Эстонии) и оценить релевантность. Кандидат заводится в `sources/candidates.yaml` (`CandidateRecord`); допущенный триажем промоутится в курируемый `meta.yaml` папки-документа через `schema.promote_candidate()`. НЕ WebFetch для тела документа — прогоняет через маленькую модель, не verbatim; годится только для метаданных.
2. **Скачивание оригинала** — через `run_pipeline.py`: acquisition-лестница `direct → official_alt → manual → archive` при WAF-блоке/мёртвом URL (см. «Слой знаний»), в `sources/{track}/{entity}/{doc-id}/raw.pdf`; sha256/канал добычи → `.state.yaml`.
3. **Конвертация** — `pdf_to_markdown.py` даёт `doc.md`; frontmatter порождается из `meta.yaml` (`schema.render_frontmatter`), не пишется вручную.
4. **Подтверждение** — Стадия 2: по тексту `doc.md` подтвердить/понизить `relevance.target_fit`; `validate_sources.py` требует `relevance` у каждой записи корпуса (гейт в CI и в `run_pipeline`).

Подробности слоя метаданных/графа/поиска — раздел «Слой знаний» ниже.

## Слой знаний: метаданные, граф, семантический поиск
Поверх собранных `.md` — слой для кросс-документного анализа (Фазы 1-4 реализованы, влиты в `main` через PR #1; проектный план и статус — `pipeline/setup/knowledge-graph-metadata/spec.md`, gitignored). Всё CPU-only, встроенное, без серверов/JVM/CUDA. Скрипты запускаются ИЗ КОРНЯ: `.venv/bin/python pipeline/scripts/<script>.py`.

- **Метаданные** — `schema.py` (слим по corpus-layout-v2, PR #4): курируемое ядро `SourceRecord` (~19 полей) в `meta.yaml` — id, `entity_id`/`track`, issuer_type, geo_scope, language (ISO 639-1, либо 639-3 там, где 639-1 нет — черногорский `cnr`; PR #5), dates, doc_type, authority, source_url, sensitivity, rights, topics, g2ai_pattern, summary, relations, relevance (Dublin Core + минимум аналитики). Операционное (sha256/acquisition/…) — отдельная модель `OperationalState` в `.state.yaml`-sidecar (машина пишет `save_state` через `fsio.atomic_write_text`, НЕ в `meta.yaml`; comment-safe); `raw_size`/`raw_mtime_ns` — stat-guard, чтобы не перечитывать raw целиком на каждом прогоне (PR #7). Инварианты раскладки (папка==id, entity_id, track; глоб. уникальность id) — единая `schema.check_layout()` (PR #8), используется и `load_records()` (raise), и `validate_sources.py` (collect); пути raw/doc.md/state ВЫВОДЯТСЯ из папки (`raw_file`/`md_file`/`state_file`). `render_frontmatter()` порождает frontmatter. `validate_sources.py` — валидатор (структура + словари + уникальность id + relations + `relevance` + инварианты папок + для `geo_scope=national` форма `entity_id` == iso2, PR #5), возвращает `(errors, records)` — потребители (`run_pipeline`/`build_graph`) переиспользуют записи без повторного обхода дерева (PR #8); ненулевой код при ошибках, гейт также в CI (PR #5).
- **Словари** — `pipeline/vocab/vocab_*.yaml` (растут органически); `g2ai_pattern` = переносимый паттерн-решение (ось кросс-странового сравнения), `topics` = тема. `jurisdictions.yaml` (EU/ASEAN) даёт рёбра `member_of`.
- **Граф** — `build_graph.py`: гетерогенный NetworkX-граф (узлы document/pattern/topic/issuer/jurisdiction; рёбра exemplifies/about/published_by/applies_to/member_of + document→document из relations). Экспорт GraphML. Запросы: `docs_by_pattern`, `docs_in_bloc`, `lineage`.
- **Поиск** — единый `pipeline/index/corpus.db` (SQLite): `corpus_index.py` (канонические ~512-токенные чанки + FTS5 `unicode61`; `build`/`search`), `vector_store.py` (векторы + брутфорс-косинус; `embed-corpus`/`vsearch`). Чанки канонические — общие для FTS и векторов. `index_meta` (key-value таблица, PR #7) хранит `corpus_fingerprint` (stat-based отпечаток корпуса — реконсиляция пересборки индекса, не in-run флаг) и `chunk_max_tokens`; пересборка чанков ДРОПАЕТ таблицу `vectors` целиком (chunk_id переиспускается без AUTOINCREMENT — старый вектор иначе молча указывал бы на чужой текст), `vectors_fingerprint:<model>` сверяется перед поиском (`VectorsStaleError` вместо неверных результатов) — стоп-гэп до будущей content-hash-инкрементальности. `corpus_index.py search` по умолчанию экранирует запрос (`sanitize_fts_query` — дефис не входит в bareword-грамматику FTS5), `--raw` — для честного FTS5-синтаксиса.
- **Эмбеддинги** — `embed.py`: `OnnxBgeEmbedder` (локальный bge-m3 int8 ONNX, CPU, ~3 мин на документ, RSS ~1.8 ГБ) + `OpenRouterEmbedder` (эталон, ключ из `.env` — см. `.env.example`). `ab_eval.py` — A/B качества (bge vs эталон). Токенизатор — `bge_tokenizer.py`; загрузка `.env` — `env.py`.
- **Оркестратор** — `run_pipeline.py`: идемпотентный end-to-end по корню `sources/` (download→convert→frontmatter→index). Реконсиляция desired-state (работа выводится из ФС: наличие `raw.*` + sha256 из `.state.yaml`, свежесть `doc.md`, синхронность frontmatter) — повторный прогон no-op, самовосстановление; пересборка индекса реконсилируется тем же принципом — по `corpus_fingerprint`, не по in-run флагу (PR #7). Ручной/неотслеженный raw (`--no-download`) усыновляется — sha256 считается и фиксируется реконсиляционным шагом, не только при автоскачивании (PR #7). Атомарная запись артефактов документа — через `fsio.py` (единая staging-политика, dot-префиксные файлы вне глобов `raw.*`, PR #6/#8). Отказ документа не рвёт батч (включая стадию планирования, PR #6); quality-gate (валидация реестра), `--dry-run`/`--force`/`--only`/`--no-download`/`--embed`/`--watch-dir`. Сознательно БЕЗ Airflow/Prefect/Dagster (серверные, оверкилл для масштаба/железа).
- **Добыча / WAF-устойчивость** — `acquisition.py` (спек `pipeline/setup/source-acquisition-ladder/spec.md`, PR #2): лестница `direct → official_alt → manual → archive` на стадии download. Детекция блока — не только HTTP-код, но и Cloudflare-отпечатки (`cf-ray`/`__cf_bm`/challenge-маркеры в теле) и content-magic, иначе `200`-challenge сохранился бы как «документ» (реальный кейс: ai.gov.ae, error 1020, заблокировал и `curl`, и автоматизированный браузер); сетевые отказы curl (DNS/connect, exit 6/7) классифицируются как `dead` и уходят в archive-ступень, а не роняют документ (PR #6). `manual` — 1-клик путь (`xdg-open` официального URL + watch-folder поверх системной папки загрузок, `xdg-user-dir DOWNLOAD`), синхронен только при `--only` (батч на нём не виснет, просто репортит и идёт дальше); кандидаты сверяются с title документа (грубый порог по словам, с мемоизацией парсинга) — посторонний PDF в папке загрузок больше не может тихо стать оригиналом; явный `--watch-dir` отключает сверку (PR #6). `archive` — снимок из Wayback CDX API для подтверждённо мёртвых URL (`fidelity: archived_snapshot`, никогда не `live`; `sensitivity: confidential` отключает archive — Wayback публикует обращение; запрос берёт РЕАЛЬНО свежайший снимок через `limit=-N`, не первые N по возрастанию, PR #6). Состояние добычи (`acquisition_method`/`fidelity`/`acquisition_checked`/`retrieved_snapshot_date` + sha256) пишется в `.state.yaml`-sidecar документа (`schema.save_state`, plain YAML — файл машиннописаный, без комментариев; `ruamel`/round-trip и `persist_acquisition_state` убраны как ненужные после разделения курируемого/операционного — corpus-layout-v2, PR #4).

- **Триаж релевантности / слой кандидатов** — `schema.py` + `validate_sources.py` (спек `pipeline/setup/source-relevance-triage/spec.md`, PR #3): гейт-ФИЛЬТР допуска источника в реестр, перед добычей. Кандидаты — в **отдельном `sources/candidates.yaml`** как `CandidateRecord` (лёгкий `extra="allow"` тип с провенансом добычи; НЕ `status:candidate` на `SourceRecord`); допуск = конверсия типа `promote_candidate() → SourceRecord`, не смена статуса. Релевантность двухстадийна (метаданные+WebSearch ДО добычи → подтверждение по тексту ПОСЛЕ конвертации), по двум осям (`agentic_g2ai` узкая / `digital_sovereignty` широкая), тир `primary`/`context`/`background`. Дата — сигнал, не гейт: `frontier_year` в `pipeline/config/triage.yaml`. Провенанс добычи остаётся в `candidates.yaml` (в `meta.yaml` НЕ копируется — corpus-layout-v2). Автоматический Стадия-1-судья — отложен (пока триаж ручной). **Генератор кандидатов (DISCOVERY)** — спроектирован чартером `pipeline/setup/discovery/architecture.md` (мультиконнекторный портфель: registry/watcher/directed-search/manual; коннектор №1 — ETO AGORA), ещё НЕ реализован.

Модель bge-m3 (~543 МБ) качается отдельно в `pipeline/models/bge-m3-onnx-int8/` (int8 ONNX, команды — в проектном плане). Тесты, зависящие от модели, помечены `@pytest.mark.model` и в CI пропускаются (`-m "not model"`). Векторы разных моделей несравнимы — на корпусе живёт одна модель за раз.

## `pipeline/scripts/pdf_to_markdown.py` — как устроен и его реальные ограничения
Полностью автоматический конвертер PDF→Markdown на `pdfplumber` (MIT, pure Python, CPU-only, ~70MB venv — сознательно НЕ marker-pdf/datalab-to, см. ниже почему). Механизмы:
- **Восстановление порядка чтения в многоколоночной вёрстке** — geometric gap-анализ по bounding-box словам (проекция покрытия по x, поиск устойчивого по высоте разрыва в центральной трети страницы), а не regex-эвристика поверх готового текста.
- **Иерархия заголовков** — по кластерам font-size конкретного документа (пересчитывается заново для каждого PDF из его собственной гистограммы), НЕ по regex на нумерации "1.1.1" — работает и для документов без явной нумерации разделов.
- **Колонтитулы/номера страниц** — частотный анализ ЦЕЛЫХ строк (не отдельных слов) в верхней/нижней полосе страницы.
- **Сноски** — отдельный (заниженный, ~65% от размера тела) порог только для крошечных надстрочных номеров-маркеров — сам текст сноски обычно почти неотличим по размеру от текста обычных врезок/кейс-стади, попытка развести их одним порогом теряла бы кейс-стади.
- **Таблицы** — `pdfplumber.find_tables()` + фильтр по АБСОЛЮТНОМУ числу непустых ячеек (не по доле — у настоящих таблиц со слитыми ячейками доля заполненности так же низка, как у шумных обломков диаграмм).
- **Диаграммы/инфографика** — НЕ пытаемся угадывать порядок чтения; детектируются по большим локальным горизонтальным разрывам между словами и помечаются в выводе как «требует ручной проверки» с сырым текстом внутри (текст не теряется, просто честно помечена неопределённость порядка).

**Известное ограничение — пороги калибровались на одном документе** (Сингапур, IMDA Model AI Governance Framework for Agentic AI v1.5, 2026-07-15) и не должны переноситься вслепую на следующие страны. *2-й прогон (Эстония, MKM «Tehisintellekti tegevuskava 2024–2026», эстонский, 47 стр., 2026-07-15) ПОДТВЕРДИЛ слабые места (не починил): диаграмм-эвристика ложно срабатывает на оглавлении и SWOT-матрицах; многостраничные таблицы фрагментируются с повтором шапок и пустыми колонками. Извлечение текста, заголовков и диакритики (õ/ä/ö/ü) — чисто.* Конкретика порогов:
- `WORD_GAP_THRESHOLD=25pt` и `TINY_MARKER_RATIO=0.65` — абсолютные числа, не относительные к кеглю конкретного документа
- Детектор колонок ловит только ОДИН разрыв (максимум 2 колонки) — 3-колоночная вёрстка не поддержана
- Заголовки различаются только по РАЗМЕРУ шрифта, не по жирности/цвету — документ с одинаковым кеглем заголовка/тела будет пропущен целиком
- Диаграммная эвристика — самое слабое место, подобрана под один конкретный флоучарт
- **Специфический риск для Японии** (в seed-списке `sources/temporary/small_states_v01.md` есть японская стратегия): `pdfplumber.extract_words()` токенизирует по пробелам — если исходный PDF на японском (CJK, без пробелов между словами), группировка в строки/колонки может не сработать; проверить наличие англоязычной версии до прогона.

**Практика на каждый новый документ:** тот же выборочный аудит, что делался для Сингапура — сверка списка заголовков, проверка таблиц на дубли/фрагментацию, чтение хотя бы одного «сложного» (многоколоночного/с диаграммой) участка целиком. Не доверять результату вслепую.

## Почему НЕ marker-pdf / НЕ Jina Reader (сравнение проведено 2026-07-15)
- **marker-pdf** (datalab-to) — GPL код + Open RAIL-M веса, тянет ~1.3GB+ CUDA-стека (torch/cudnn/nccl) даже на CPU-only задаче. На рабочей машине физически несовместим: GeForce 940MX (Maxwell, compute capability 5.2) выпала из поддержки CUDA 13.x/актуального PyTorch — установку пришлось прервать и полностью вычистить.
- **Jina Reader** (r.jina.ai) — по умолчанию PDF.js (не LLM), но при живом тесте на реальном гос. PDF (тот же сингапурский документ) получил 403 Forbidden — WAF издателя (tech.gov.sg) заблокировал IP Jina, хотя прямой `curl` с той же машины скачал файл без проблем. Системный риск, специфичный именно для корпуса государственных документов — у многих гос. сайтов WAF банит дата-центровые/прокси IP.
- `pdfplumber` — ноль GPU-зависимостей, работает локально, не проксирует запросы через сторонние серверы (важно, т.к. документы могут быть чувствительны к раннему разглашению до публикации итогового пакета предложений).

## Известные факты о рабочей машине
